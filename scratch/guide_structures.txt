[Title] PSAT User Guide
[Heading 1] Path Safety Assessment Tool (PSAT) User Guide
[Normal] Welcome to PSAT. The tool helps you create projects from geotagged survey images, review coded path segments, calculate risk, and test treatments.
[Heading 2] 1. Getting Started
[Heading 3] Open the project list
[Normal] Use the Home page to browse all projects. You can:
[List Bullet] search by project name or road name
[List Bullet] filter by tag
[List Bullet] sort by verification, autocode progress, and last modified time
[List Bullet] select one or more projects for coding, path analysis, or treatment work
[Heading 3] Create a project
[Normal] Use Create Project when you want to build a new project from the source image folders in `in/`.
[Normal] You can create a project in ways:
[List Number] select one source folder directly
[List Number] draw a polygon or click a planning area on the map and create a project from multiple selected roads
[Normal] When creating a project:
[List Bullet] enter a project name without underscores
[List Bullet] add tags if needed
[List Bullet] upload images into a source folder if the folder is missing
[List Bullet] check that selected roads are marked as available before you create the project
[Heading 3] Navigate between workflows
[Normal] From the project list, you can send selected projects to:
[List Bullet] Coding for detailed review and saving
[List Bullet] Analyse Projects for multi-project filtering, charts, and exports
[List Bullet] Treatment Application for before/after scenario testing
[Heading 3] Use Help and GIS Layers
[List Bullet] use the Help button for the in-app user and developer guides
[List Bullet] use View GIS Layers from the sidebar when you need to inspect the GIS data behind the coding rules
[Heading 2] 2. Coding Page
[Normal] The Coding page is the main review workspace. It can open one or more selected projects in a combined session.
[Heading 3] Main layout
[Normal] The page keeps three views in sync:
[List Bullet] the current segment image
[List Bullet] the attributes table
[List Bullet] the segment map
[Normal] Selecting a segment in one area updates the others.
[Heading 3] Auto-code options
[Normal] PSAT supports several auto-code paths:
[List Bullet] CV auto-code from the image
[List Bullet] GIS auto-code from the segment geometry
[List Bullet] bulk auto-code across selected rows or the full project
[List Bullet] per-attribute auto-code in workflows that target only certain
[Normal] Autocode updates are tracked, and the Segments Autocoded counter is updated in the project metadata.
[Heading 3] Manual review
[Normal] You can override any coded value directly in the table. The page also shows:
[List Bullet] score updates for the selected segment
[List Bullet] a validation summary comparing against the stored baseline
[List Bullet] field-source provenance for auto-coded changes
[Heading 3] Details and GIS context
[Normal] For supported attributes, the page can show extra spatial detail such as:
[List Bullet] nearby GIS layers around the current segment
[List Bullet] curvature visualization
[List Bullet] width visualization
[List Bullet] grade or gradient details when profile data is available
[Heading 3] Save and progress tracking
[Normal] After review:
[List Bullet] save your attribute edits to persist them and recalculate scores
[List Bullet] update the Segments Verified counter as you complete manual checks
[Heading 2] 3. Map View & Analysis
[Normal] Map views appear in several PSAT workflows, especially Coding, Path Analysis, Treatment, and GIS Layers.
[Heading 3] Segment map in Coding
[Normal] On the Coding page, the map helps you:
[List Bullet] click a segment to select it
[List Bullet] see risk-band coloring for saved scoring results
[List Bullet] view nearby GIS context for the current point
[List Bullet] inspect the spatial position of the current segment
[Heading 3] Risk colors
[Normal] Segments are color-coded by their highest relevant risk band so you can quickly spot high-risk areas.
[Heading 3] GIS context
[Normal] Depending on the page and workflow, PSAT can show nearby GIS information such as:
[List Bullet] paths and crossings
[List Bullet] bus stops and MRT-related features
[List Bullet] road-name and planning-area context
[List Bullet] width curvature analysis overlays
[Heading 3] GIS Layers page
[Normal] Use the dedicated GIS Layers page when you want to inspect the raw shapefile layers themselves rather than just the map context around a segment.
[Normal] That page lets you preview available layers, review their metadata, and update GIS files when needed.
[Heading 2] 4. Path Analysis
[Normal] The Path Analysis page is the multi-project analysis workspace.
[Heading 3] Select projects
[Normal] You can select one or more projects to analyze together. The search box also matches source-road names, so you can find projects even when the project title and road name differ.
[Heading 3] Filter before loading
[Normal] Before loading projects, you can filter by:
[List Bullet] project or road name
[List Bullet] tag
[List Bullet] date created range
[List Bullet] last updated range
[Heading 3] Analyze loaded data
[Normal] After loading projects, you can:
[List Bullet] choose up to five attributes to focus on
[List Bullet] apply segment-level filters
[List Bullet] view attribute distributions and aggregated score-band summaries
[List Bullet] inspect filtered segments on the map and in the table
[Heading 3] Export
[Normal] The page supports two main exports:
[List Bullet] Download Table for the current filtered table as CSV
[List Bullet] Download Images for the current filtered images as a ZIP file
[Heading 3] Session continuity
[Normal] Your selections and filters are kept for the browser session, so navigating away and returning does not immediately clear the analysis setup.
[Heading 2] 5. Treatment Application
[Normal] The Treatment Application page lets you test safety improvements on one or more cycling path projects before committing to any changes. You can simulate what would happen if you applied standard CycleRAP interventions, preview the impact on risk scores, and compare before-and-after results across the entire project.
[Heading 3] Step 1: Find and load your projects
[Normal] When you first open Treatment Application, you will see a project picker — the same list you use on the Home page. Use it to choose which projects to work on:
[List Bullet] Type a project name or road name in the Search box to filter the list.
[List Bullet] Use the Filter by tag dropdown to narrow down by tag (e.g. Pre, Post, or a custom location tag).
[List Bullet] The Verification Status column shows a green tick (✅) for fully verified projects and an hourglass (⏳) for projects still in progress.
[List Bullet] Click a row (or tick its checkbox) to select it. You can select multiple projects at once, or use the Select All row to select every project currently visible.
[List Bullet] Click the pencil icon in the Actions column to rename a project or change its tags before loading.
[List Bullet] Once you have selected the projects you want, click Load Project to open the treatment workspace.
[Heading 3] Step 2: Understanding the treatment workspace layout
[Normal] After loading, the workspace is divided into several areas:
[List Bullet] Project tabs (shown at the top when you loaded more than one project) — click a tab to jump to the first segment of that project.
[List Bullet] Before Treatment and After Treatment maps — a side-by-side pair of maps at the top of the page, showing every segment colour-coded by risk level. The left map always shows the original scores; the right map updates as you apply treatments.
[List Bullet] A three-column panel below the maps — Treatment Options on the left, the segment image in the middle, and the scores and attributes on the right.
[List Bullet] An Overall Treatment Analysis section at the bottom with pie charts comparing the project-wide risk distribution before and after all treatments you have applied.
[Normal] The page number shown at the top right (e.g. 3 / 120) tells you which segment you are currently viewing out of the total across all loaded projects. Use the Previous and Next buttons, or type a segment number directly into the input box, to navigate.
[Heading 3] Step 3: Choose how to view treatments — By Segment or By Treatment
[Normal] The Treatment Options panel on the left has a dropdown that lets you switch between two views:
[List Bullet] By Segment — shows only the treatments that are relevant to the segment you are currently viewing. Each treatment is ranked by how much it would reduce the risk score for that specific segment (shown as "Score drop: X.X"). Use this view when you want to focus on improving one segment at a time.
[List Bullet] By Treatment — shows all treatments that are applicable to at least one segment across the entire loaded project. Each treatment shows the percentage of segments it would improve (e.g. "Improves 12.5% of segments"). Treatments are ranked from most to least effective. Use this view when you want to apply a single intervention type across many segments at once.
[Heading 3] Step 4: Select and preview treatments
[Normal] Tick the checkbox next to one or more treatments to select them. Selected treatments are highlighted in blue. You can also use the All button at the bottom of the panel to select every applicable treatment at once, or the Clear button to deselect all.
[Normal] As soon as you tick a treatment, the Scores card on the right will show a live preview of what the risk scores would look like if you applied those treatments — without actually saving anything yet. The preview appears with a light background to indicate it is not yet confirmed. You can select multiple treatments to see their combined effect.
[Normal] The attributes panel also has a Show Post-Treatment toggle. Switch this on to see exactly which attribute values would change and how, highlighted in the table.
[Heading 3] Step 5: Apply treatments
[Normal] When you are satisfied with your selection, click Apply (N) at the bottom of the Treatment Options panel — where N is the number of treatments you have selected.
[List Bullet] In By Segment mode, Apply saves the selected treatments for the current segment only. The treatment card turns green with a tick (✓) to confirm it has been recorded, and the After Treatment map updates to reflect the change.
[List Bullet] In By Treatment mode, Apply triggers a confirmation dialog listing each selected treatment and asking whether you want to apply it to all eligible segments across the project. Click Confirm to proceed. This is useful for applying a single policy improvement (such as "Install street lighting") everywhere it is needed in one action.
[Normal] If you change your mind about an applied treatment on the current segment, click the Reset button that appears after a treatment has been applied. This removes all treatments from that segment and restores the original scores.
[Heading 3] Step 6: Read the segment image, scores, and attributes
[Normal] The middle column shows the street-level photograph for the current segment. Use Previous and Next to move between segments while staying on the same page. If no image is available for a segment, the panel will be empty.
[Normal] The right column shows two panels:
[List Bullet] Segment Scores — displays the four CycleRAP risk scores for the current segment: Vehicle-Bicycle (VB), Bicycle-Bicycle (BB), Single-Bicycle (SB), and Bicycle-Pedestrian (BP). Each score is shown alongside its risk band (Low, Medium, High, or Extreme) and the attributes that contributed most to it. When a treatment preview is active, the scores update to reflect the expected change.
[List Bullet] Attributes panel — lists all coded attribute values for the current segment. Toggle Show Post-Treatment on to highlight which fields would be changed by the selected or applied treatments (they are marked in colour with the label "Modified by treatment"). Toggle it off to see the original coded values.
[Heading 3] Step 7: Before and After maps
[Normal] The two maps at the top of the page give you a project-wide view of how treatments are changing risk levels across all segments:
[List Bullet] Before Treatment map — always shows the original risk-band colouring for every segment (green = Low, yellow = Medium, orange = High, purple = Extreme).
[List Bullet] After Treatment map — updates in real time as you apply treatments. Segments that have been treated and whose risk band has improved will appear in a lighter colour. Click any segment on either map to navigate to it.
[Heading 3] Step 8: Generate AI visualisations of proposed improvements
[Normal] The Treatment Options panel includes two clipboard buttons that help you create visual representations of proposed improvements using an AI image tool (such as ChatGPT image generation):
[List Bullet] Copy prompt — copies a ready-to-use text prompt describing the selected treatments in plain language, tailored to the Singapore cycling infrastructure context. Paste this prompt into your AI image tool alongside the segment photograph to generate a before-and-after visualisation.
[List Bullet] Copy image — copies the current segment photograph directly to your clipboard. Paste it into the same AI tool together with the prompt. Both buttons work together: copy the image first, then paste it into the AI tool, then paste the prompt text.
[Normal] The prompt is automatically adapted to whichever treatments you have selected or applied for the current segment, so you always get a description that matches what you are proposing.
[Heading 3] Step 9: Review the Overall Treatment Analysis
[Normal] Scroll to the bottom of the page to see the Overall Treatment Analysis section. It shows two rows of five pie charts — one for each crash type (Overall Risk Level, Bicycle-Bicycle, Single-Bicycle, Bicycle-Pedestrian, and Vehicle-Bicycle):
[List Bullet] Before Treatment row — shows how the full project is currently distributed across the four risk bands.
[List Bullet] After Treatment row — shows the updated distribution once all treatments you have applied are taken into account. If you have only treated a subset of segments, the remaining untreated segments retain their original scores.
[Normal] Use these charts to get a quick sense of whether the treatments you have applied make a meaningful dent in the overall risk profile of the project.
[Heading 3] Tips and common workflows
[List Bullet] Pre and Post tagging — if you are working with a before-and-after survey pair, tag one project "Pre" and the other "Post" on the Home page. Load both into Treatment Application together to compare their score distributions side by side using the project tabs.
[List Bullet] Applying a single improvement across a whole project — switch to By Treatment view, find the treatment you want (e.g. "Install street lighting"), tick it, and click Apply. Confirm in the dialog. The After Treatment map will update immediately to show the effect across all eligible segments.
[List Bullet] Undoing all changes — if you want to start fresh, navigate to each segment that has been treated and click Reset. The Overall Treatment Analysis will update accordingly.
[List Bullet] Checking what was applied — treatments that have already been saved are shown with a green background and a tick (✓) in the list. If a treatment appears greyed out and untickable, it means it has already been applied to the current segment.
[Heading 1] GIS Layer Management Guide
[Normal] PSAT uses external GIS shapefiles and GeoJSON layers to provide context for path coding and risk analysis. Use the GIS Layer Management tool to keep these datasets up to date.
[Heading 2] 1. Viewing GIS Layers
[Normal] Click the View GIS Layers button in the sidebar to open the GIS Layers dashboard. Here you can:
[List Bullet] Browse all current layers grouped by category (e.g., `area_type`, `bus_lane`).
[List Bullet] View metadata for each layer, including the geometry type and feature count.
[List Bullet] **Required Columns**: Inspect the mandatory column names needed for PSAT to process the layer. The number in parentheses, e.g., `LU_DESC (1)`, indicates the expected column index in the source data.
[Heading 2] 2. Adding a New GIS Layer
[Normal] Use the Add GIS Layer workflow to upload entirely new datasets.
[List Bullet] **New Category**: You must provide a name for the category folder (e.g., `school_zones`). Every upload is assigned to a specific category.
[List Bullet] **File Upload**: Drag and drop your GIS files. For shapefiles, ensure you upload all companion files together (`.shp`, `.shx`, `.dbf`, `.prj`).
[List Bullet] **Preview**: Once uploaded, you can preview the geometry on the map before finalizing.
[Heading 2] 3. Replacing an Existing GIS Layer
[Normal] Use the Replace GIS Layer workflow when you have updated data for an existing layer.
[List Bullet] **Filterable Search**: Use the searchable inputs to quickly find the folder and specific layer you wish to replace. Simply type a few letters (e.g., "bus") to filter the list.
[List Bullet] **Safety Checks**: PSAT performs compatibility checks to ensure the new file has the same required columns as the original.
[List Bullet] **Warnings**: If differences are found in the column structure, the system will warn you before overwriting the old data.

[Normal] The Path Safety Assessment Tool (PSAT) is an internal web application for assessing cycling and active-mobility corridors with the CycleRAP v2.11 methodology. It combines image-driven coding, GIS lookups, manual review, scoring, and treatment testing in one workflow.

[List Bullet] calculate **BB**, **BP**, **SB**, and **VB** risk scores plus overall risk bands
[List Bullet] analyze one or more projects together with filters, charts, and exports
[List Bullet] test treatments and compare before/after risk outcomes

[Heading 1] Architecture Overview
[Normal] PSAT is a two-container application orchestrated by Docker Compose. The frontend is a React SPA served by nginx, and the backend is a Flask API that owns project storage, GIS lookups, CV inference, scoring, and treatment logic. There is no database; the file system under `data/`, `in/`, and `backend/shapefiles/` is the source of truth.

[Heading 2] High-level flow
[List Number] The user creates or selects a project in the frontend.
[List Number] The frontend calls `/api/projects/*` and `/api/shapefiles/*` as needed.
[List Number] The backend loads project metadata, geodata, snapshot CSVs, shapefiles, and models lazily.
[List Number] Coding, analysis, and treatment flows mutate the latest snapshot and persist derived artifacts such as scores, baselines, and autocode metadata.
[List Number] The frontend rehydrates the updated data through normal REST reads.

[Heading 3] Lazy initialization
[Normal] `get_ctx()` in `projects/routes.py` lazily initializes and caches:
[List Bullet] `project_manager` over `data/`
[List Bullet] serializer / attribute-mapping data
[List Bullet] the legacy CycleRAP interface helpers where still needed
[Normal] CV models are loaded separately by `_ensure_models_ready()`. Initialization failures are memoized so repeated requests fail fast with HTTP 503 rather than repeatedly blocking on model load.

[Normal] That CSV is optional, but when present it lets the create-project map show which intersecting roads actually have local image folders.

[Heading 2] Project creation pipeline
[Normal] Project creation still starts from geotagged source images in `in/`, but it now supports both single-folder and multi-folder/polygon flows.
[Normal] Backend steps:
[List Number] enumerate one or more source folders
[List Number] extract EXIF GPS data from images
[List Number] optionally filter nodes by a selection polygon
[List Number] geocode and sample points at the project-creation stage
[List Number] convert sampled points to LineStrings
[List Number] copy images into the project image store
[List Number] create metadata, geodata, and the initial dated snapshot
[Normal] When multiple source folders are merged, copied image names are namespaced so duplicate filenames do not collide.
[Heading 2] Multi-project aggregation pattern
[Normal] Several frontend pages load multiple projects into one combined view. The usual pattern is:
[List Bullet] concatenate attributes, results, and geodata into global arrays
[List Bullet] keep a `projectMap` of project name, start index, and count
[List Bullet] resolve UI actions back to `{ projectName, localIndex }` when saving, deleting, or treating a row
[Normal] This pattern is especially important in coding, treatment, and path-analysis views.
[Heading 2] Shapefile management architecture
[Normal] The GIS Layers page is backed by the `gis_layers` blueprint. That blueprint:
[List Bullet] scans `backend/shapefiles/`
[List Bullet] groups layers by category (subdirectory)
[List Bullet] can return a selected shapefile as WGS84 GeoJSON
[List Bullet] validates uploaded ZIPs and replacement files
[List Bullet] supports upload, preview, replace, and delete operations
[Normal] This means the same layer inventory drives both:
[List Bullet] the GIS Layers admin page
[List Bullet] the backend GIS mapping logic used during coding
[Heading 2] Key design choices
[Heading 3] API-only frontend communication
[Normal] The frontend never reads the filesystem directly. Every read/write goes through REST endpoints, which keeps all storage rules and migration logic on the backend.
[Heading 3] Native scoring path
[Normal] The current score endpoint uses `calculate_cyclerap_score_native()` instead of Excel COM. Legacy Excel helpers still exist, but native Python scoring is the main path.
[Heading 3] CRS handling
[Normal] Source image GPS is read in WGS84 (`EPSG:4326`). Stored project geodata is written in Singapore SVY21 (`EPSG:3414`) so geometric distance and width/curvature operations behave sensibly.
[Heading 3] Thread safety
[Normal] Model loading is protected by a lock and cached error state, which avoids repeated slow failures under concurrent requests.

[Heading 1] CV / ML Pipeline
[Normal] PSAT uses a suite of YOLO-based computer-vision models to automatically infer CycleRAP attribute values from street-level photographs. This process is called auto-coding.
[Normal] ---
[Heading 2] Overview
[Normal] ```
[Normal] Street-level photograph (.jpg)
[Normal] │
[Normal] ▼
[List Number] Path Segmentation (path_seg.pt)
[Normal] └─ Detects: path (0), road (1), buffer zone, etc.
[Normal] │
[Normal] ├── No path detected → manual review required
[Normal] │
[Normal] └── Path detected
[Normal] │
[Normal] ├─ 2. Off-Road Bicycle Classifier (off_road_bicycle_path.pt)
[Normal] │     └─ Is this an off-road bicycle path?
[Normal] │
[Normal] ├─ 3. Adjacent Road Lane Classifier (adj_road_lane.pt)
[Normal] │     └─ Road within 0–1m? 1–3m? Neither?
[Normal] │
[Normal] ├─ 4. Fixed Obstacle Detector (LTA_FIXEDOBSTACLE_BEST_2.pt)
[Normal] │     └─ Obstacles, bus stops, delineation markers
[Normal] │
[Normal] ├─ 5. Development Access Classifier (DevelopmentAccess_last_150epochs.pt)
[Normal] │     └─ Driveway / property access present?
[Normal] │
[Normal] └─ 6. Delineation Classifier (LTA_Dill_4_Best.pt)
[Normal] └─ Road markings / lane delineation present?
[List Number] Road Classification Model (RoadClassification_best.pt)
[Normal] └─ Determines facility type context
[Normal] │
[Normal] ▼
[Normal] Attribute dictionary {field: coded_value}
[Normal] │
[Normal] ▼
[Normal] Stored in attributes.csv (merged with any existing values)
[Normal] ```

[Normal] |---|---|---|
[Normal] | `path_seg.pt` | Segmentation | Path (0), road (1), buffer zone, zebra crossing, etc. |

[Heading 3] Model Loading
[Normal] Models are loaded once at first use by `CycleRAP_Coding_Helper.initialise(model_dir)` in `prediction.py`. The backend searches for `path_seg.pt` in these locations (in order):
[List Number] `MODEL_DIR` environment variable (if set)
[List Number] `backend/model/`
[List Number] `backend/models/`
[List Number] Adjacent `model/` or `models/` relative to the source path
[Normal] If no model directory is found, the backend raises a `RuntimeError` and returns HTTP 503 for all subsequent CV requests.
[Heading 3] Replacing or Updating a Model
[Normal] When a retrained or improved `.pt` file becomes available, follow these steps:
[List Number] **Drop the new file into `backend/models/`**, using the **exact same filename** as the model it replaces (e.g. `path_seg.pt`). The filename is hardcoded in `prediction.py` — changing it will break loading.
[Normal] ```
[Normal] backend/models/
[Normal] ├── path_seg.pt                          ← replace this
[Normal] ├── off_road_bicycle_path.pt
[Normal] ├── adj_road_lane.pt
[Normal] ├── LTA_FIXEDOBSTACLE_BEST_2.pt
[Normal] ├── DevelopmentAccess_last_150epochs.pt
[Normal] ├── LTA_Dill_4_Best.pt
[Normal] └── RoadClassification_best.pt
[Normal] ```
[List Number] **Rebuild the Docker image** so the new file is copied into the container:
[Normal] ```bash
[Normal] docker compose up --build
[Normal] ```
[List Number] **Verify the model loaded correctly** by checking the backend logs on startup and sending a test auto-code request. The models are loaded lazily on the first CV request, not at boot.
[Normal] > If you need to rename the new file (e.g. the new model has a different name): update the corresponding filename string in `prediction.py` inside `CycleRAP_Coding_Helper.initialise()`, then rebuild.
[Normal] > Storing multiple versions: Keep old `.pt` files archived on the SSD. Only the file with the exact expected name in `backend/models/` will be used by the app.
[Normal] All inference logic lives in the `CycleRAP_Coding_Helper` class (static methods only).
[Heading 3] Entry Point: `autocode(image_path)`
[Normal] Returns a dictionary mapping every `Attributes.Fields` field to a coded integer value (or `None` if the field cannot be determined from the image).
[Heading 3] Step 1 — Path Segmentation
[Normal] ```python
[Normal] seg_results = cls.path_segmentation_model.predict(image_path)
[Normal] result = cls._filter_segmentation_results(seg_results[0], confidence_threshold=0.5)
[Normal] ```
[Normal] The segmentation model produces per-pixel class masks. Detected class IDs:
[List Bullet] `0` = path / cycle facility
[List Bullet] `1` = road
[Normal] If no path (class `0`) is detected, a warning is logged and manual review is flagged.
[Heading 3] Step 2 — Light Segregation (Default)
[Normal] If a path is detected, `Light Segregation` is set to `Present (1)` by default.
[Heading 3] Step 3 — Adjacent Road (if road class `1` is present)
[Normal] ```python
[Normal] adj_result = cls.adj_road_lanes_classifier.predict(cropped)[0]
[Normal] pred_adj_class = int(adj_result.probs.top1)
[Normal] ```
[Normal] The cropped path bounding box is passed to the adjacency classifier:
[List Bullet] Class `1` → road is within **0–1m**: sets `Adjacent Road Lane 0-1m = Present`, `1-3m = Not Present`
[List Bullet] Class `2` → road is within **1–3m**: sets `Adjacent Road Lane 0-1m = Not Present`, `1-3m = Present`
[List Bullet] Class `0` or low confidence → uncertain, logged for manual review
[Normal] Confidence threshold: `0.8`. Results below this threshold default to class `0` (uncertain).
[Heading 3] Step 4 — Off-Road Bicycle Path Classification
[Normal] ```python
[Normal] cls_result = cls.off_road_bicycle_classifier.predict(cropped)[0]
[Normal] ```
[Normal] Confidence threshold: `0.8`. If classified as an off-road bicycle path:
[List Bullet] Sets `Facility Type = Off-Road Bicycle Path (3)`
[List Bullet] Sets `Delineation = Present (1)`
[List Bullet] If multiple path indices detected: sets `Adjacent Sidewalk 0-1m = Present`
[Heading 3] Step 5 — Facility Type Decision (via Fixed Obstacle Model)
[Normal] The fixed obstacle model is run on the full image. A pixel-level segmentation map is built:
[Normal] ```
[Normal] pathway_area    = pixels with label 0
[Normal] road_area       = pixels with label 8
[Normal] left_road_area  = road in left half of image
[Normal] right_road_area = road in right half of image
[Normal] ```
[Normal] Decision logic:
[Normal] | Condition | Facility Type |
[Normal] |---|---|
[Normal] | Bus stop or bus-related obstacle detected (labels `1`, `4`) | Sidewalk |
[Normal] | Bottom half >90% pathway AND pathway > road | Multi-Use Path |
[Normal] | Road confined to one side (>80%) | Sidewalk |
[Normal] | Only pathway, no road | Sidewalk |
[Normal] | Default (road visible across full width) | Mixed Traffic Road Lane |
[Heading 3] Step 6 — Fixed Obstacle & Delineation Detection
[Normal] A second pass with `LTA_FIXEDOBSTACLE_BEST_2.pt` checks whether detected obstacle masks overlap the path mask:
[Normal] ```python
[Normal] FIXED_OBSTACLE_RELEVANT_LABELS = {1, 2, 4, 6, 7}
[Normal] NON_FIXED_OBSTACLE_LABEL = 9
[Normal] DELINEATION_RELEVANT_LABELS = {0, 1, 2, 5}
[Normal] ```
[List Bullet] If any relevant obstacle intersects the path: sets `Fixed Obstacle on Facility = Present`
[List Bullet] If label `9` (non-fixed) intersects path: sets `Non-Fixed Obstacle on Facility = Present`
[List Bullet] Delineation markers on path: sets `Delineation = Present`

[Heading 1] Scoring Logic
[Normal] PSAT implements the CycleRAP v2.11 risk scoring algorithm as a pure Python module (`cyclerap_scoring.py`). No Excel, VBA macros, or Windows-only dependencies are required.
[Normal] ---
[Heading 2] Overview
[Normal] CycleRAP produces four independent risk scores for each road segment, each representing a distinct crash scenario:
[Normal] | Score | Crash type | Description |
[Normal] |---|---|---|
[Normal] | BB | Bicyclist–Bicyclist | Conflict between cyclists (and with light vehicles) |
[Normal] | BP | Bicyclist–Pedestrian | Conflict between a cyclist and a pedestrian |
[Normal] | SB | Single Bicyclist | Cyclist departing facility and striking a fixed/severe hazard |
[Normal] | VB | Vehicle–Bicyclist | Conflict between a motor vehicle and a cyclist |
[Normal] Each score is a continuous real number. Scores are then classified into one of four risk bands.
[Normal] An Overall Risk Level (the sum of all four scores) and an Overall Risk Level Band (the maximum of the four component bands) are also computed.
[Normal] ---
[Heading 2] Risk Bands
[Heading 3] BB, BP, SB
[Normal] | Band | Label | Score range |
[Normal] |---|---|---|
[Normal] | 1 | Low | < 5 |
[Normal] | 2 | Medium | 5 – 10 |
[Normal] | 3 | High | 10 – 20 |
[Normal] | 4 | Extreme | > 20 |
[Heading 3] VB
[Normal] | Band | Label | Score range |
[Normal] |---|---|---|
[Normal] | 1 | Low | < 10 |
[Normal] | 2 | Medium | 10 – 25 |
[Normal] | 3 | High | 25 – 60 |
[Normal] | 4 | Extreme | > 60 |
[Heading 3] Overall Risk Level Band
[Normal] Equal to the maximum band across BB, BP, SB, and VB for that segment. This means a single extreme sub-score elevates the overall band to Extreme regardless of the others.
[Normal] ---
[Heading 2] Algorithm: Component Formulas
[Normal] The algorithm builds four intermediate components (`CM3`, `CM16`, `CM25`, `CM40`) and then combines them into the final scores.
[Heading 3] CM3 — Main Cycling Environment Risk
[Normal] ```
[Normal] CM3 = (product of CU factors) ^ (1 + CQ_sum × 0.1)
[Normal] ```
[Normal] CU factors (continuous risk multipliers):
[Normal] | Factor | Attribute | Effect |
[Normal] |---|---|---|
[Normal] | Loose/slippery surface | `Loose or slippery surface` | 1.5 if Present |
[Normal] | Delineation | `Delineation` | 1.2 if Not Present |
[Normal] | Facility width | `Facility Width per Direction` | 1.8/1.5/1.0 for Very Narrow/Narrow/Wide |
[Normal] | Flow direction | `Flow Direction` | 1.5 if Two Way |
[Normal] | Width restriction | `Width Restriction` | 1.2 if Present |
[Normal] | Grade | `Grade` | 1.2 if ≥5 Degrees |
[Normal] | Cargo bikes | `Observed proportion of cargo bikes` | 1.2 if Moderate to high |
[Normal] | Pedestrian flow | `Peak pedestrian flow` | 1.2 if Low; 1.5 if Moderate to high |
[Normal] | Bicycle flow | `Peak bicycle/LV traffic flow` | 1.2 if Moderate to high |
[Normal] | Speed differential | `Bicycle/LV speed differential` | 1.2 if ≥10 km/h |
[Normal] | Curvature | `Curvature` | 1.5 if Sharp Turn Present |
[Normal] | Street lighting | `Street Lighting` | 1.2 if Not Present |
[Normal] CQ triggers (condition flags that increase the exponent):
[Normal] | Trigger | Attribute | Fires when |
[Normal] |---|---|---|
[Normal] | Surface deformation | `Major Surface Deformation or Drain Opening` | Present |
[Normal] | Fixed obstacle | `Fixed Obstacle on Facility` | Present |
[Normal] | Non-fixed obstacle | `Non-Fixed Obstacle on Facility` | Present |
[Normal] | Adjacent parking 0–1m | `Adjacent Vehicle Parking 0-1m` | Present |
[Normal] | Intersection crossing | `Intersection or Road Crossing` | Present |
[Normal] | Property access | `Property Access` | Present |
[Normal] | Intersecting facility | `Intersecting Bicycle Facility` | Present |
[Normal] | Pedestrian crossing | `Pedestrian Crossing` | Present |
[Heading 3] CM16 — Departure and Fall Scenarios
[Normal] Only fires if at least one of: Loose/slippery surface, Grade ≥5°, or Curvature = Sharp Turn.
[Heading 3] CM25 — Speed-Related Incidents
[Normal] Only fires if at least one of: Tram/train rails present, Surface deformation present.
[Heading 3] CM40 — Vehicle Interaction
[Normal] Only fires if at least one of: Intersection crossing, Property access, Adjacent road 0–1m, Adjacent road 1–3m, Facility type = Mixed Traffic, Intersection approach = Shared.
[Heading 3] Final Score Combination
[Normal] The four components combine into final scores as follows:
[Normal] ```
[Normal] cj3  = CM3 + CM16   # combined conflict exposure
[Normal] cj25 = CM25         # fall hazard modifier
[Normal] cj40 = CM40         # shared space conflict modifier
[Normal] BB = cj3 × (bicycle_speed × grade × speed_differential × cargo_bike factors)
[Normal] BP = BB_component × sidewalk_condition
[Normal] + BB_component × pedestrian_condition
[Normal] (each term is 0 if the condition is absent)
[Normal] SB = cj3 × severity_factors                  [departure gate — 0 if no departure condition]
[Normal] + (cj3 + cj25) × fall_severity_factors    [fall gate — 0 if no fall condition]
[Normal] VB = (cj3 + cj25) × factors   [fall/conflict — 0 if condition absent]
[Normal] + cj3 × factors             [departure — 0 if condition absent]
[Normal] + cj40 ÷ 3                  [intersection term 1 — 0 if condition absent]
[Normal] + cj40 ÷ 3                  [intersection term 2 — 0 if condition absent]
[Normal] + cj40 ÷ 3                  [intersection term 3 — 0 if condition absent]
[Normal] ```
[Normal] Key takeaway: BB and BP are driven entirely by CM3 + CM16. SB additionally incorporates CM25 for fall scenarios. VB is the only score that uses CM40 (shared-space and intersection conflicts).
[Normal] ---
[Heading 2] Worked Example: CM3
[Normal] The following example traces a single segment through the CM3 calculation by hand.
[Normal] Segment attributes:
[Normal] | Attribute | Value | Code |
[Normal] |---|---|---|
[Normal] | Loose or slippery surface | Not Present | 2 |
[Normal] | Delineation | Not Present | 2 |
[Normal] | Facility Width per Direction | Narrow | 2 |
[Normal] | Flow Direction | One Way | 1 |
[Normal] | Width Restriction | Not Present | 2 |
[Normal] | Grade | < 5 Degrees | 1 |
[Normal] | Observed proportion of cargo bikes | Low | 1 |
[Normal] | Peak pedestrian flow | None | 1 |
[Normal] | Peak bicycle/LV traffic flow | Low | 1 |
[Normal] | Bicycle/LV speed differential | < 10 km/h | 1 |
[Normal] | Curvature | Sharp Turn Present | 1 |
[Normal] | Street Lighting | Present | 1 |
[Normal] | All CQ-trigger fields | Not Present / N/A | 2 |
[Normal] Step 1 — Build the CU product:
[Normal] ```
[Normal] CU = 1.0  (loose surface: Not Present → 1.0)
[Normal] × 1.2  (delineation:  Not Present → 1.2)
[Normal] × 1.5  (width:        Narrow      → 1.5)
[Normal] × 1.0  (flow:         One Way     → 1.0)
[Normal] × 1.0  (width restr:  Not Present → 1.0)
[Normal] × 1.0  (grade:        <5°         → 1.0)
[Normal] × 1.0  (cargo bikes:  Low         → 1.0)
[Normal] × 1.0  (ped flow:     None        → 1.0)
[Normal] × 1.0  (bike flow:    Low         → 1.0)
[Normal] × 1.0  (speed diff:   <10 km/h    → 1.0)
[Normal] × 1.5  (curvature:    Sharp Turn  → 1.5)
[Normal] × 1.0  (lighting:     Present     → 1.0)
[Normal] = 1.8
[Normal] ```
[Normal] Step 2 — Count the CQ triggers:
[Normal] All eight CQ-trigger fields (surface deformation, fixed obstacle, non-fixed obstacle, adjacent parking 0–1m, intersection crossing, property access, intersecting facility, pedestrian crossing) are coded as Not Present. CQ_sum = 0.
[Normal] Step 3 — Apply the formula:
[Normal] ```
[Normal] CM3 = CU_product ^ (1 + CQ_sum × 0.1)
[Normal] = 1.8 ^ (1 + 0 × 0.1)
[Normal] = 1.8 ^ 1.0
[Normal] = 1.8
[Normal] ```
[Normal] Now add one CQ trigger — set Intersection or Road Crossing = Present (1):
[List Bullet] CQ_sum = 1 (intersection_crossing fires)
[Normal] ```
[Normal] CM3 = 1.8 ^ (1 + 1 × 0.1)
[Normal] = 1.8 ^ 1.1
[Normal] ≈ 1.91
[Normal] ```
[Normal] The exponent increases by 0.1 per active CQ trigger, compounding the base risk. Eight simultaneous triggers would raise the exponent to 1.8, producing `1.8^1.8 ≈ 2.66`.
[Normal] Road traffic volume is converted to a risk factor using a stepped lookup table:
[Normal] | AADT threshold | Risk factor |
[Normal] |---|---|
[Normal] | < 100 | 0.25 |
[Normal] | 100 – 499 | 0.25 |
[Normal] | 500 – 1499 | 0.50 |
[Normal] | 1500 – 2499 | 0.75 |
[Normal] | 2500 – 4999 | 1.00 |
[Normal] | 5000 – 7499 | 1.07 |
[Normal] | 7500 – 9999 | 1.14 |
[Normal] | 10000 – 12499 | 1.23 |
[Normal] | 12500 – 14999 | 1.31 |
[Normal] | 15000 – 17499 | 1.40 |
[Normal] | 17500 – 19999 | 1.50 |
[Normal] | 20000 – 22499 | 1.61 |
[Normal] | 22500 – 24999 | 1.72 |
[Normal] | 25000 – 29999 | 1.84 |
[Normal] | 30000 – 34999 | 1.97 |
[Normal] | 35000 – 39999 | 2.10 |
[Normal] | ≥ 40000 | 2.25 |
[Heading 3] Road Speed Risk Factor
[Normal] Operating speed is converted via a sigmoid formula:
[Normal] ```
[Normal] lookup_speed = round(speed) - 1
[Normal] risk = 1 + 27.82 / (1 + exp(5.84 − 0.091 × lookup_speed))
[Normal] ```
[Normal] > Speed is first rounded to the nearest integer, then decremented by 1 before the sigmoid is applied. Returns 1.0 if speed ≤ 1.

[Heading 2] Attribute Fields Reference

[Heading 2] Treatment Logic
[Normal] The 25 predefined treatments (defined in `routes.py`) each have:
[List Bullet] **Triggers:** one or more sets of `{field: [allowed_values]}` conditions. If any trigger set matches, the treatment is applicable.
[List Bullet] **Effects:** `{field: new_value}` pairs applied to the segment's attributes before re-scoring.
[Normal] Treatments are evaluated by `apply_treatments`, `apply_all_treatments`, and `preview_treatments` endpoints. The projected score change (before/after) is returned so the user can see the improvement.

[Heading 1] Common Issues
[Normal] This page focuses on the issues that show up most often in current PSAT workflows, especially around Docker setup, polygon-based project creation, and GIS-layer management.
[Heading 2] Docker and startup
[Heading 3] Docker Desktop is not running
[Normal] Symptom: `docker compose up --build` fails immediately with a Docker pipe / connection error.
[Normal] Fix: Start Docker Desktop and wait for it to report that the engine is running, then retry.
[Heading 3] Port 80 or 8000 is already in use
[Normal] Symptom: The stack starts partially, but the frontend or backend is unreachable.
[Normal] Fix: Stop the conflicting service or change the port mappings in `docker-compose.yml`.
[Heading 3] Backend health check fails after build
[Normal] Symptom: `http://localhost:8000/api/ping` does not respond.
[Normal] Fix: Check `docker compose logs backend` first. The most common causes are:
[List Bullet] missing `backend/models/`
[List Bullet] missing `backend/shapefiles/`
[List Bullet] a bad local edit that breaks imports
[Heading 2] Models and shapefiles
[Heading 3] Auto-code fails because models are missing
[Normal] Symptom: CV auto-code returns 503 or logs show `missing path_seg.pt`.
[Normal] Fix: Copy the shared model files into `backend/models/` and rebuild.
[Heading 3] GIS auto-code or GIS Layers page is empty
[Normal] Symptom: GIS-assisted coding returns nothing, or the GIS Layers page has no usable layers.
[Normal] Fix: Confirm the shapefile tree exists under `backend/shapefiles/` and contains the expected layer files, not just empty folders.
[Heading 2] `in/` folder and source images
[Heading 3] `in/` was created by Docker instead of manually
[Normal] Symptom: File writes into `in/` fail or behave strangely.
[Normal] Fix: Stop Docker, recreate `in/` manually from the host, then restart.
[Heading 3] Images are directly under `in/` instead of inside subfolders
[Normal] Symptom: The folder dropdown is empty or project creation fails.
[Normal] Fix: Put images inside subdirectories of `in/`, one folder per source road / survey batch.
[Heading 3] Images have no GPS EXIF data
[Normal] Symptom: Project creation fails, or you get zero usable segments.
[Normal] Fix: The source images must contain GPS EXIF metadata. Screenshots, stripped images, and many exported image sets will not work.
[Heading 2] Polygon and road selection
[Heading 3] Polygon selection returns planning areas instead of roads
[Normal] Symptom: The map selection tool only returns high-level planning-area names, or the road list looks incomplete.
[Normal] Cause: The backend falls back when it cannot get better road matches from `road_reference.csv` and road-name shapefiles.
[Normal] Fix:
[List Number] Make sure `backend/shapefiles/road_reference.csv` exists.
[List Number] Regenerate it after adding or renaming folders in `in/`:
[Normal] ```bash
[Normal] cd backend
[Normal] python generate_road_reference.py
[Normal] ```
[List Number] Confirm the road-name shapefiles are present under `backend/shapefiles/`.
[Heading 3] Selected roads are marked unavailable
[Normal] Symptom: The Create Project page shows roads, but some are flagged as missing and block creation.
[Normal] Fix: Those roads do not currently have matching local folders under `in/`. Either:
[List Bullet] add the missing source folders and images
[List Bullet] or deselect the unavailable roads before creating the project
[Heading 3] Create Project says no geotagged images were found inside the polygon
[Normal] Symptom: `POST /api/projects/folders` returns a 400 error about no geotagged images inside the selected polygon.
[Normal] Cause: The selected roads may exist, but none of their sampled GPS points survived the polygon filter.
[Normal] Fix:
[List Bullet] redraw a larger polygon
[List Bullet] verify the images in those folders have GPS EXIF data
[List Bullet] regenerate `road_reference.csv` if the source folders changed recently
[Heading 2] Project metadata and search
[Heading 3] Searching by road name does not return the expected project
[Normal] Symptom: The Projects, Treatment, or Path Analysis page does not surface a project when you search for a road.
[Normal] Cause: Newer projects store `source_folders` explicitly. Older projects are reconstructed from image namespaces on a best-effort basis, which may be incomplete if images predate the multi-folder naming convention.
[Normal] Fix: For older projects, check the actual project metadata or recreate the project if road-level provenance is critical.
[Heading 3] Project name cannot contain underscores
[Normal] Symptom: Project creation returns `Project name cannot contain underscores (_)`.
[Normal] Fix: Use spaces, hyphens, or camel case instead.
[Heading 2] Scoring and coding
[Heading 3] Scores do not update after save
[Normal] Symptom: The attributes save succeeds but the displayed scores look stale.
[Normal] Fix: Check backend logs for scoring errors. `PUT /api/projects/<name>/attributes` recalculates and persists scores automatically.
[Heading 3] First auto-code request is very slow
[Normal] Expected behavior: The first CV request loads the YOLO models into memory. Subsequent requests are much faster.
[Heading 2] Documentation drift
[Heading 3] The repository docs were updated, but the Help page still shows old content
[Normal] Cause: The Help page reads mirrored markdown from `frontend/public/docs/`, not directly from `docs/`.
[Normal] Fix: Sync both locations whenever docs change.
[Heading 2] Data recovery
[Heading 3] Projects disappear after container restart
[Normal] Symptom: `docker compose down` followed by `up` appears to lose projects.
[Normal] Fix: Confirm that `./data:/app/data` is still present in `docker-compose.yml` and that the host `data/` directory still exists.
[Heading 3] A project was deleted accidentally
[Normal] Cause: Project deletion removes the project directory recursively.
[Normal] Fix: Restore it from a filesystem backup if available. There is no soft-delete layer.
[Heading 1] Contributing
[Normal] These notes are intentionally practical rather than process-heavy. The codebase is large, stateful, and file-backed, so the safest contributions are small, explicit, and well-validated.
[Heading 2] Working style
[List Bullet] keep changes focused on one behavior or one documentation topic at a time
[List Bullet] avoid mixing unrelated cleanup into feature or bug-fix work
[List Bullet] prefer fixes at the owning layer instead of UI-only or data-only workarounds
[List Bullet] preserve existing file formats and naming conventions unless the task requires a migration
[Heading 2] Validation expectations
[Normal] Run the narrowest useful validation for the area you changed.
[Normal] Typical checks:
[List Bullet] backend route or service changes: targeted Flask smoke test or Python import/compile check
[List Bullet] frontend code changes: TypeScript build or targeted page validation
[List Bullet] documentation changes: sync both `docs/` and `frontend/public/docs/`, then do a stale-content sweep
[Heading 2] Documentation rule
[Normal] If you update repository docs in `docs/`, update the mirrored Help-page copies in `frontend/public/docs/` as part of the same change.
[Heading 2] Project-data safety
